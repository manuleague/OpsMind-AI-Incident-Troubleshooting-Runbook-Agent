# RB-HTTP-001 Application Returning 502 or 503

## Retrieval Keywords

http 502, http 503, bad gateway, service unavailable, unhealthy backend, app gateway, load balancer, upstream timeout, checkout failure

## Symptoms

- Users receive HTTP 502 Bad Gateway or HTTP 503 Service Unavailable.
- Gateway or load balancer reports unhealthy backend targets.
- Error rate increases after deployment, configuration change, or dependency outage.
- Synthetic login, checkout, or API transaction fails.

## Diagnostic Steps

1. Confirm whether errors are global or limited to a route, region, tenant, or backend pool.
2. Check deployment history, configuration changes, feature flags, and traffic shifts.
3. Inspect gateway backend health, upstream timeout metrics, and connection reset counts.
4. Review application logs for startup failures, dependency errors, connection pool exhaustion, or missing configuration.
5. Validate database, cache, identity provider, queue, and downstream API health.

## Likely Causes

- Bad deployment or incompatible configuration.
- Backend instances failing health checks.
- Gateway timeout caused by slow dependency or saturated app workers.
- Dependency outage causing the application to fail startup or request handling.

## Safe Remediation

- Prepare rollback if a deployment correlates with the start time and rollback compatibility is confirmed.
- Restart only unhealthy instances when the platform supports low-blast-radius restart.
- Disable a new feature flag if evidence shows it caused the failure and owner approval is recorded.
- Do not restart all production instances or roll back without incident commander approval.

## Validation Checks

- HTTP 5xx rate returns to baseline.
- Gateway backend health is green.
- Synthetic transaction succeeds.
- Application logs stop showing startup or dependency errors.

## Escalation Triggers

- Rollback, production restart, dependency failover, or data migration decision is required.
- Payment, authentication, or customer-facing critical path is impacted.
- The same error persists after one safe mitigation attempt.
